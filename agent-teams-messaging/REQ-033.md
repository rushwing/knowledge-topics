# Goal

在 REQ-032 统一 Envelope 基础上，引入消息生命周期目录结构，
实现 Claim 原子性机制，防止 cron 重启或多消费者场景下的重复消费。

**这是防止 Agent 任务被重复触发的核心防护机制。**

# In Scope

- `scripts/pandas-heartbeat.sh`：`inbox_init()` 创建四级子目录（pending/claimed/done/failed）
- `scripts/pandas-heartbeat.sh`：`inbox_write_v2()` 默认写入 `for-{target}/pending/`
- `scripts/pandas-heartbeat.sh`：`inbox_read_pandas()` 实现 Claim 原子性
  1. 扫描 `for-pandas/pending/*.md`
  2. `mv pending/$f claimed/$f`（原子性；mv 失败则 skip）
  3. 执行 handler 处理
  4. 处理成功：`mv claimed/$f done/$f`
  5. handler 异常退出（set -e 捕获）：`mv claimed/$f failed/$f`，追加错误摘要到文件末尾
- `menglan-heartbeat.sh` 中的 inbox_read：同步适配新目录结构（仍使用原有处理逻辑）
- `huahua-heartbeat.sh` 中的 inbox_read：同步适配新目录结构（仍使用原有处理逻辑）
- `scripts/harness.sh`：`inbox_init` 子命令同步创建四级目录

# Out of Scope

- 文件命名规范变更（时间戳格式） → REQ-034
- expired 状态 / watchdog 超时检测 → REQ-035 或单独 REQ
- UI 层消息状态展示

# Acceptance Criteria

- `inbox_init()` 创建 `for-pandas/{pending,claimed,done,failed}/` 及 menglan / huahua 对应目录
- `inbox_write_v2()` 写入 `for-{target}/pending/`（而非扁平目录）
- `inbox_read_pandas()` 在读取每条消息前先 `mv pending/$f claimed/$f`
- 若 mv 失败（文件已不存在），直接 skip（不报错）
- handler 成功后 `mv claimed/$f done/$f`
- handler 失败（任何原因）后 `mv claimed/$f failed/$f` + 追加错误摘要到文件末尾
- 新旧两种目录结构兼容（扁平目录中残存的旧文件仍可被读取处理）
- `npm run build` 通过
- `npm test` 通过，新增 Claim 机制单元测试

## Claim 机制核心实现参考

```bash
inbox_read_pandas() {
  local inbox_dir="${INBOX_ROOT}/for-pandas"
  local pending_dir="${inbox_dir}/pending"
  local claimed_dir="${inbox_dir}/claimed"
  local done_dir="${inbox_dir}/done"
  local failed_dir="${inbox_dir}/failed"

  # 兼容旧扁平目录：同时扫描 pending/ 和根目录
  local scan_dirs=("$pending_dir")
  [[ -d "$inbox_dir" ]] && scan_dirs+=("$inbox_dir")  # 旧文件可能在根目录

  local count=0
  for scan_dir in "${scan_dirs[@]}"; do
    [[ -d "$scan_dir" ]] || continue
    for msg_file in "${scan_dir}"/*.md; do
      [[ -f "$msg_file" ]] || continue

      local base_name
      base_name="$(basename "$msg_file")"
      local claimed_file="${claimed_dir}/${base_name}"

      # 原子 Claim：mv 失败则 skip（文件已被他人取走）
      mv "$msg_file" "$claimed_file" 2>/dev/null || {
        info "skip（已被 claim）: ${base_name}"
        continue
      }

      count=$(( count + 1 ))
      info "claimed: ${base_name}"

      # 处理消息（失败则 mv 到 failed/）
      if _process_message "$claimed_file"; then
        mv "$claimed_file" "${done_dir}/${base_name}"
        info "done: ${base_name}"
      else
        echo "" >> "$claimed_file"
        echo "# Processing failed at $(date -u +%Y-%m-%dT%H:%M:%SZ)" >> "$claimed_file"
        mv "$claimed_file" "${failed_dir}/${base_name}"
        warn "failed: ${base_name}"
      fi
    done
  done

  [[ $count -eq 0 ]] && info "inbox/for-pandas/ 无新消息"
}
```

## 幂等性保证

- Claim 是 mv 原子操作：同一文件只能被一个进程 mv 成功
- done/ 中的文件不会被重复扫描（不在 pending/ 目录下）
- 若 cron 在 claimed/ 阶段重启，下次启动会扫到 claimed/ 中残存文件 — 建议在 `inbox_init()` 中检测 claimed/ 残存文件并 warn（不自动处理，等人工决策）
