# Goal

在现有 `scripts/pandas-heartbeat.sh` 的 inbox 机制上，新增统一消息 Envelope 语义层。

**不改变目录结构，不改变传输机制（cron + 文件系统）。**
只在 frontmatter 层面做扩展，是最小侵入的语义升级。

这是 ATM（Agent Teams Messaging）协议的 P0 Sprint，为后续 Correlation / Claim 机制打基础。

# In Scope

- `scripts/pandas-heartbeat.sh`：新增 `inbox_write_v2()` 函数，支持新 Envelope 字段
- `scripts/pandas-heartbeat.sh`：`inbox_read_pandas()` 兼容新旧格式（按字段是否存在判断）
- 新 Envelope 必填字段：`message_id`、`type`（request|response|notification）、`from`、`to`、`created_at`、`thread_id`、`correlation_id`
- type=request 附加字段：`action`、`objective`、`response_required`
- type=response 附加字段：`in_reply_to`、`status`（completed|partial|blocked|failed|rejected|deferred）、`summary`
- type=notification 附加字段：`event_type`、`severity`（info|warn|action-required）
- 旧有 `inbox_write()` 函数保留，标注 `@deprecated`，内部调用 `inbox_write_v2()` 以保证向后兼容
- 现有调用 `inbox_write()` 的地方逐步迁移（本 REQ 内完成 pandas-heartbeat.sh 内部调用迁移）
- `message_id` 生成规则：`msg_{from}_{yyyymmddHHMMSS}_{rand4}`
- `thread_id` 生成规则：`thread_{req_id}_{epoch}`（由 Pandas 在首次发出 request 时创建）
- `correlation_id` 生成规则：`corr_{req_id}_{epoch}`

# Out of Scope

- 目录结构变更（pending/claimed/done/failed 子目录 → REQ-033）
- Claim 原子性机制 → REQ-033
- 文件命名规范变更（新时间戳格式） → REQ-034
- Delegation 结构化字段强制校验 → REQ-035
- menglan-heartbeat.sh / huahua-heartbeat.sh 中的 inbox 调用迁移（各自单独 REQ）

# Acceptance Criteria

- `inbox_write_v2()` 存在，签名：`<target> <type> <action_or_event> <thread_id> <correlation_id> <in_reply_to> <priority> <response_required> <payload_file>`
- `inbox_write_v2()` 生成的文件 frontmatter 包含所有必填 Envelope 字段
- `inbox_write()` 调用 `inbox_write_v2()` 并传入对应参数，旧调用不破坏
- `inbox_read_pandas()` 能解析新 Envelope 格式（按 `type` 字段值路由 handler）
- `inbox_read_pandas()` 仍能解析旧格式（`type: implement` 等旧值映射为 `action`，`type` 字段缺失时按旧逻辑路由）
- 新增对 `type: notification` 的处理分支（severity=action-required 时触发 tg_notify）
- `npm run build` 通过
- `npm test` 通过，新增覆盖新格式解析和旧格式兼容的单元测试

## inbox_write_v2() 参考实现

```bash
# inbox_write_v2 <target> <type> <action_or_event> <thread_id> <correlation_id>
#                <in_reply_to> <priority> <response_required> <payload_file>
#
# payload_file: 临时文件，包含 type-specific 附加字段 + Markdown body
# 示例 payload_file（type=request）：
#   objective: 实现 REQ-030 Claim 机制
#   scope: scripts/pandas-heartbeat.sh
#   expected_output: PR 已开，tests 通过
#   done_criteria: mv 原子化，done/failed 目录正确归类
#   ---
#   （Markdown body）
inbox_write_v2() {
  local target="$1"
  local type="$2"
  local action_or_event="$3"
  local thread_id="$4"
  local correlation_id="$5"
  local in_reply_to="${6:-}"
  local priority="${7:-P2}"
  local response_required="${8:-false}"
  local payload_file="${9:-}"

  local now msg_id date_str slug filename target_dir
  now="$(date -u +%Y%m%d%H%M%S)"
  msg_id="msg_${AGENT_ORCHESTRATOR:-pandas}_${now}_$(cat /dev/urandom | tr -dc 'a-z0-9' | head -c 4)"
  date_str="$(date -u +%Y-%m-%dT%H:%M:%SZ)"
  slug="$(echo "${type}-${action_or_event}" | tr '[:upper:]' '[:lower:]' | tr '_' '-')"
  filename="${date_str//:/-}__${type}__$(whoami)_to_${target}__${correlation_id:-notset}.md"
  target_dir="${INBOX_ROOT}/for-${target}"
  mkdir -p "$target_dir"

  {
    echo "---"
    echo "message_id: ${msg_id}"
    echo "type: ${type}"
    echo "from: ${AGENT_ORCHESTRATOR:-pandas}"
    echo "to: ${target}"
    echo "created_at: ${date_str}"
    echo "thread_id: ${thread_id}"
    echo "correlation_id: ${correlation_id}"
    echo "priority: ${priority}"
    # type-specific
    case "$type" in
      request)
        echo "action: ${action_or_event}"
        echo "response_required: ${response_required}"
        ;;
      response)
        echo "in_reply_to: ${in_reply_to}"
        ;;
      notification)
        echo "event_type: ${action_or_event}"
        ;;
    esac
    echo "---"
    # payload（type-specific 字段 + body）
    [[ -n "$payload_file" && -f "$payload_file" ]] && cat "$payload_file"
  } > "${target_dir}/${filename}"

  info "inbox_write_v2 → for-${target}/${filename} [${type}/${action_or_event}]"
}
```

## 旧格式兼容逻辑（inbox_read_pandas 扩展）

读取消息时，先检查 `type` 字段是否为新 Enum（request|response|notification）：
- 是新格式 → 按 `type` 路由
  - type=request：按 `action` 字段路由（implement / tc_design / bugfix / fix_review / escalate / clarify）
  - type=response：按 `status` 字段路由
  - type=notification：按 `severity` 触发对应处理（action-required → tg_notify）
- 是旧格式（type 值不在新 Enum 内）→ 按原有 case 逻辑路由（implement / tc_design / dev_complete 等）
- type 字段缺失 → warn + 跳过
