# Goal

完成 ATM 协议的 P2 Sprint，落实两个剩余规范：

1. **Delegation 结构化**：request 消息的任务委派内容必须结构化，
   不允许纯自然语言 body 作为任务规格
2. **文件命名规范**：新格式文件名携带路由信息，
   人和 agent 直接从文件名读取上下文

# In Scope

- `scripts/pandas-heartbeat.sh`：`inbox_write_v2()` 在 type=request 时校验 payload_file 中
  必须包含 `objective` / `scope` / `expected_output` / `done_criteria` 四个字段
  - 校验失败：warn 日志 + 仍写入文件（不阻断），但在 envelope 中标记 `delegation_incomplete: true`
- 新文件命名规范：
  ```
  {timestamp}_{type}_{from}_to_{to}_{corr_or_evt}.md
  示例：
  2026-03-19T17-47-00Z_request_pandas_to_menglan_corr_REQ-030_1710867000.md
  2026-03-19T18-02-00Z_response_menglan_to_pandas_corr_REQ-030_1710867000.md
  2026-03-19T18-05-00Z_notification_menglan_to_pandas_evt_artifact_created.md
  ```
- `inbox_read_pandas()` 兼容新旧两种文件命名（旧格式：`YYYY-MM-DD-pandas-{slug}-*.md`）
- `menglan-heartbeat.sh` / `huahua-heartbeat.sh` 更新文件命名生成逻辑
- `context_summary` 字段（选填）：限制 500 字，超长自动截断 + warn
- `references` 字段（选填）：list of `{path, type}` 对象，type ∈ req | tc | bug | source

# Out of Scope

- open-workhorse UI 的 thread 链路视图（可单独 REQ 实现）
- 历史消息（done/）文件名批量重命名

# Acceptance Criteria

- `inbox_write_v2()` type=request 时，payload_file 缺少 objective / scope / expected_output / done_criteria 任一字段 → warn 日志 + `delegation_incomplete: true` 写入 envelope
- 新命名格式文件名可被 `inbox_read_pandas()` 正常解析
- 旧格式文件名（`YYYY-MM-DD-pandas-...`）仍可被正常解析（兼容 6 个月）
- `npm run build` 通过
- `npm test` 通过，覆盖委派字段校验和新旧文件名兼容场景

## 为什么 delegation_incomplete 不阻断写入

严格校验（拒绝写入）会在过渡期破坏现有调用。
标记 `delegation_incomplete: true` 的设计是：
- 让消息仍然可以被投递和处理
- 让接收方 agent 知道这条 request 的任务规格可能不完整
- 逐步引导调用方补充结构化字段

当所有调用方都已迁移后，可将策略改为拒绝写入（在 REQ-035 后的后续 REQ 中执行）。
